# 从经验到技能：OpenClaw 知识蒸馏完全指南

## 背景

OpenClaw Agent 在日常任务执行中持续产生"经验"——这些经验存在于对话历史、决策轨迹、错误恢复路径中。问题是：它们只存在于隐式状态，无法跨会话复用。每当新的会话启动，Agent 需要重新从零开始摸索。

**知识蒸馏**（Knowledge Distillation）正是解决这一问题的系统方法：把散乱的经验碎片提炼为结构化的 Skill 模块，让 Agent 在任何会话中都能调用经过验证的最佳实践。

本文面向 OpenClaw 进阶用户、Agent 开发者以及希望建立知识库的系统管理员，提供一份可直接落地的蒸馏实践指南。

## 推荐方案：三种蒸馏对象与六步法

### 三种待蒸馏对象

不是所有经验都值得蒸馏，也不是所有经验都该用同一种方式蒸馏。根据输入形态，蒸馏对象分为三类：

| 对象 | 形态 | 来源 | 典型示例 |
|------|------|------|---------|
| **知识** | 事实、判断标准、偏好规则 | 对话中反复验证的信息 | "张总偏好 BLUF 格式报告" |
| **流程** | 多步骤操作序列、决策分支 | 重复任务的共同执行路径 | 每日调研工作流（采集→分析→推送） |
| **轨迹** | 历史行为模式、错误恢复路径 | sessions 历史、cron 执行日志 | npm install 失败后的恢复序列 |

### 知识蒸馏：从隐式到显式

核心问题：**"这个 Agent 是如何知道 X 的？"**

```
对话/执行历史
    ↓ 模式识别
重复出现的判断/偏好
    ↓ 显式化
SKILL.md 中的规则 或 references/ 中的文档
```

**提炼检查清单**：
- [ ] 这个信息在对话中出现过 ≥2 次吗？
- [ ] 这是用户的明确偏好还是偶然表述？
- [ ] 将它写入 Skill vs 临时查询，哪个更高效？

**知识类型与存放位置对照**：

| 知识类型 | 存放位置 | 示例 |
|---------|---------|------|
| 事实性 (Facts) | `references/*.md` | 用户公司基本信息、联系人偏好 |
| 判断规则 (Rules) | `SKILL.md` 正文 | "涉及 `rm -rf` 必须二次确认" |
| 偏好模式 (Preferences) | 独立偏好库文件 | "用户说'不要'即永久规则" |
| 领域知识 (Domain) | `references/` | 金融/法律等行业知识库 |

### 流程蒸馏：从经验到剧本

核心问题：**"做这件事的标准步骤是什么？"**

```
多个相似任务的执行轨迹
    ↓ 比对共性，提取变量
抽象工作流（保留关键决策点）
    ↓ 参数化
SKILL.md 工作流 + 必要的 scripts/
```

**流程标准化三层次**：

```
Level 1: 文字指导（高自由度）
  → SKILL.md 中用自然语言描述流程
  → 适用于：判断空间大、上下文敏感的流程

Level 2: 伪代码/脚本框架（中等自由度）
  → SKILL.md 中含参数化脚本模板
  → 适用于：有标准模式但允许参数变化的流程

Level 3: 确定性脚本（低自由度）
  → scripts/ 中包含完整可执行脚本
  → 适用于：易出错、多步骤、必须精确执行的流程
```

**流程识别信号**：
- 同一任务被执行 ≥3 次且步骤相似
- 用户说"按上次的方式做"
- 任务有清晰的输入→处理→输出结构

### 轨迹蒸馏：从历史到模式

核心问题：**"Agent 在这件事上通常怎么做的？"**

```
sessions 历史 / cron 执行日志
    ↓ 事件序列分析
常见行为序列 + 异常恢复路径
    ↓ 归纳为规则
SKILL.md 的决策树 / 错误处理章节
```

**轨迹挖掘技术**：

| 技术 | 输入 | 输出 |
|------|------|------|
| 频繁序列挖掘 | 工具调用序列 | 常见执行模式 |
| 错误归因分析 | 错误日志 + 恢复操作 | 错误处理知识库 |
| 决策点提取 | 对话分支点 | SKILL.md 决策树 |
| 时间模式分析 | cron 执行时间/结果 | 定时任务规则 |

## 方案对比

### 蒸馏难度与优先级

| 蒸馏对象 | 难度 | 推荐优先级 | 理由 |
|---------|------|-----------|------|
| 知识 | ⭐⭐ 中 | **最高** | 最容易识别和验证，产出稳定 |
| 流程 | ⭐⭐⭐ 高 | **次高** | 需要多轨迹比对，但收益明显 |
| 轨迹 | ⭐⭐⭐⭐ 很难 | **视需求** | 挖掘成本高，仅针对高频错误场景 |

### 三种蒸馏路径对比

```
知识蒸馏 → references/ + SKILL.md 规则
           ✅ 成本低、收益快、可直接复用
           ❌ 依赖重复出现

流程蒸馏 → SKILL.md 工作流 + scripts/
           ✅ 标准化程度高、大幅降低 token 消耗
           ❌ 需要足够多的轨迹样本

轨迹蒸馏 → SKILL.md 决策树 + troubleshooting 章节
           ✅ 解决真实痛点（错误恢复）
           ❌ 样本难获取、模式难验证
```

## 实施步骤：六步蒸馏法

### Step 1：触发蒸馏 — 何时开始

**启动信号**（满足任一即评估）：

- 同一错误被纠正 ≥3 次
- 同一任务被执行 ≥3 次且流程相似
- 用户明确说"记住……"
- 发现了有价值的错误恢复路径
- 新建工作流超过 5 步

**数据采集命令**：

```bash
# 对话历史采集
sessions_history(sessionKey, limit=50)

# Cron 执行记录
cat ~/.openclaw/cron/jobs.json

# 共享上下文
ls ~/.openclaw/shared/context/recent/

# 每日记忆日志
cat memory/YYYY-MM-DD.md
```

### Step 2：模式分析 — 提炼什么

分析四个维度，回答四个问题：

| 维度 | 核心问题 | 产出 |
|------|---------|------|
| 频率 | 这件事多久发生一次？ | 优先级判断（值不值得蒸馏） |
| 变异性 | 每次做法有何不同？ | 确定自由度层次（Level 1/2/3） |
| 依赖性 | 依赖哪些工具/文件/变量？ | 确定 `scripts/` 需求 |
| 边界 | 什么情况下不能这样做？ | 安全规则写入 SKILL.md |

### Step 3：知识编码 — 如何表达

**四项原则**：

1. **先规则，后步骤** — 规则比步骤更难获取，先固化判断逻辑
2. **用具体示例锚定抽象规则** — `"当 X 时用 Y"` 优于 `"视情况处理"`
3. **标注置信度** — `"[高置信度]"` `"[待验证]"` 便于后续维护
4. **区分知识与流程** — 知识放 `references/`，流程放 `SKILL.md`

**编码检查**：
- [ ] Codex 能从这个 Skill 独立完成相关任务吗？
- [ ] Skill 触发词足够精准吗？（不重不漏）
- [ ] `SKILL.md` 主体不超过 500 行？

### Step 4：Skill 生成 — 产出结构

**Skill 标准 Anatomy**：

```
skill-name/
├── SKILL.md              ← 触发器 + 核心指令（必需）
├── scripts/              ← 可执行脚本（可选）
│   └── *.py / *.sh
├── references/           ← 参考文档（可选）
│   └── *.md
└── assets/               ← 模板/静态资源（可选）
    └── *.*
```

**已有技能迭代**：
1. 找到现有最接近的 Skill
2. 在其 `references/` 下新增专题文件
3. 或在 `SKILL.md` 中扩展章节

**全新技能初始化**：
```bash
# 使用官方脚手架工具
python3 init_skill.py --name my-new-skill
cd my-new-skill
# 编辑 SKILL.md（核心）
# 建立 references/（知识库）
# 编写 scripts/（确定性代码）
# 测试验证
```

### Step 5：验证回路 — 如何确认

| 验证类型 | 执行方式 | 通过标准 |
|---------|---------|---------|
| 功能验证 | 用 Skill 执行真实任务 | 产出正确 |
| 边界验证 | 在不安全场景测试 | 正确拒绝 |
| 效率验证 | 对比 Skill 使用前后的 token 消耗 | 降低 ≥30% |
| 回归验证 | 用历史轨迹重放 | 行为一致 |

### Step 6：持续蒸馏 — 如何维护

**触发维护的条件**：
- Skill 执行出错 → 立即追加错误处理
- 用户纠正 Skill 行为 → 视为 corrections log 存入独立文件
- 同一轨迹重复出现 ≥2 次 → 升级为确定性规则

**Skill 生命周期演进**：

```
初创期: 高自由度文字指令，观察执行情况
  ↓ 提炼
稳定期: 提炼为 scripts/ 脚本，覆盖高频路径
  ↓ 积累
成熟期: 建立 corrections.md，处理边缘情况
```

## 实操案例

### 案例 1：知识蒸馏 — 用户安全偏好

**来源**：对话中多次出现"禁止 `rm -rf`"的安全纠正

**蒸馏过程**：

```
对话历史 → 识别重复安全规则 → 写入 skill 安全章节
```

**产出**（写入 Skill `SKILL.md`）：

```markdown
## 安全边界

- 禁止对用户数据执行 `rm -rf`
- 删除文件一律用 `trash` 或 `mv ~/.Trash/`
- 执行 destructive command 前必须确认
- 涉及系统级删除操作需用户二次授权
```

### 案例 2：流程蒸馏 — 每日调研工作流

**来源**：连续多日执行的 Cron 任务（采集→调研→分析→推送）

**工作流图**：

```
[22:00] GitHub 数据采集 ──→ [23:35] 深度调研 ──→ [23:50] 缺口扫描
                                                                ↓
[Agent 通知] ◀── [00:10] 推送通知 ◀── [00:05] 文档更新 ◀──┘
```

**产出**（`SKILL.md`）：

```markdown
## 每日调研工作流

### 时间触发
- 22:00 触发 GitHub 数据采集 (`scripts/daily-sync.sh`)
- 23:35 触发深度调研 (`scripts/deep-research.sh`)
- 23:50 触发缺口扫描 (`scripts/gap-scan.sh`)
- 00:05 触发文档更新 (`scripts/doc-update.sh`)
- 00:10 推送通知 (`scripts/notify.sh`)

### 参数
- SEARCH_KEYWORDS: 每日调研关键词（来自配置）
- RESEARCH_DEPTH: 浅/深（默认深）
- OUTPUT_FORMAT: markdown/json（默认 markdown）

### 依赖
- GitHub CLI (gh)
- Python 3.9+
- 网络访问权限
```

### 案例 3：轨迹蒸馏 — 错误恢复模式

**来源**：Agent 多次遇到 npm install 失败并成功恢复

**产出**（`SKILL.md` troubleshooting 章节）：

```markdown
## 常见错误恢复

| 错误特征 | 恢复命令 |
|---------|---------|
| npm install 失败（网络/缓存） | `npm cache clean --force && npm install` |
| 端口冲突导致服务启动失败 | `openclaw config set gateway.port 18790` |
| JSON 配置文件损坏 | `python3 -m json.tool ~/.openclaw/openclaw.json > /dev/null && echo "OK"` |
| 依赖权限不足 | `sudo chown -R $(whoami) ~/.openclaw/` |

## 决策树

```
npm install 报错？
  ├─ EACCES → 权限不足 → sudo 或 --prefix
  ├─ ECONNREFUSED → 网络问题 → 检查 VPN/代理
  └─ EINTEGRITY → 缓存损坏 → npm cache clean
```

## 注意事项

### 避免常见陷阱

1. **过早蒸馏** — 只出现 1-2 次的模式不要急着蒸馏，等置信度提升
2. **过度蒸馏** — 不是所有经验都值得结构化，模糊规则比僵化规则更有用
3. **知识冗余** — 如果一个事实已在 SKILL.md 中出现，就不要再进 references/ 放一份副本
4. **忽视边界** — 规则必须同时说明"在 X 情况下适用"和"在 Y 情况下不适用"

### 工具缺口须知

当前 OpenClaw 在蒸馏支持上存在以下已知缺口：

| 缺口 | 影响 | 临时方案 |
|------|------|---------|
| 无自动轨迹分析脚本 | 轨迹挖掘依赖人工 | 定期手工审查 sessions 历史 |
| corrections 未结构化 | 偏好纠正无法积累 | 手动维护 `corrections.md` |
| Skill 版本管理缺失 | 多版本无法追踪 | 在 SKILL.md 手动加 `version` 字段 |

### 蒸馏优先级建议

```
第一步：将高频判断规则（安全/偏好）写入现有 Skill
第二步：将重复执行的流程固化为 scripts/
第三步：为高频错误建立 troubleshooting 章节
第四步：定期（每周）审查 corrections.md 并合并到 Skill
```
